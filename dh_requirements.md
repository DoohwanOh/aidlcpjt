# Requirements Document: 원스탑전환 고객동의 서비스 구축

## Intent Analysis

| 항목 | 분석 |
|---|---|
| **User Request** | 원스탑 전환을 위한 고객 동의 웹화면 구축 |
| **Request Type** | New Project |
| **Scope Estimate** | Multiple Components (프론트엔드 + 백엔드 + 외부 연동) |
| **Complexity Estimate** | Moderate (외부 시스템 연동, 본인인증, KTOA 연동) |
| **Requirements Depth** | Standard |

---

## 1. 프로젝트 개요

### 1.1 배경
- 원스톱 전환 서비스는 KTOA에서 도입한 시스템
- 기존 5단계(일반가입→신규신청→개통→해지접수→해지완료)를
  3단계(신규가입+해지신청→개통후 자동해지)로 간소화
- 타 통신사로 인터넷 전환 가입을 원하는 고객에게 **이전 통신사(기존 가입 통신사)**가 MMS로 동의 URL 발송
- 고객이 웹화면에서 전환 서비스 내용을 확인하고, 본인인증 완료 후 **기존 통신사의 인터넷 해지를 신청**

### 1.2 서비스 흐름
1. 고객이 신규 통신사에 인터넷 전환 가입 신청 (별도 채널)
2. **이전 통신사**가 고객에게 MMS 발송 (해지 동의 URL 포함)
3. 고객이 MMS 내 URL 접속 (유효기간: 15분)
4. 원스탑전환 서비스 안내 화면 확인
5. 전환 서비스 동의 + 개인정보 수집/이용 동의
6. SMS 본인인증 처리 (팝업 방식)
7. **기존 통신사 인터넷 해지 신청** (원스톱 전환 6단계 중 ② 해지 접수)
8. 완료 안내 화면

### 1.3 웹화면 담당 범위
- 원스톱 전환 6단계 중 **② 해지 접수** 단계를 담당 (이전 통신사 측 고객 동의/인증/해지 신청)
- ①(신규가입), ③~⑥(해지처리요청, 해지완상확인, 개통처리, 장비수거)는 다른 시스템에서 처리

## 2. Functional Requirements

### FR-01: MMS URL 접속 화면
- MMS에 포함된 URL로 고객이 접속하는 랜딩 페이지
- URL 유효기간: 발송 후 **15분**
- 만료된 URL 접속 시 안내 메시지 표시
- 모바일 웹 전용 (MMS 발송 기반)

### FR-02: 서비스 안내 화면
- 원스탑전환 서비스에 대한 안내 문구 표시
- 안내 문구는 백엔드 API에서 동적으로 조회 (버전 관리)

### FR-03: 고객 동의 처리
- 동의 항목:
  - 원스탑전환 서비스 이용 동의
  - 개인정보 수집/이용 동의
- 각 동의 항목별 전문 표시 및 체크박스
- 모든 필수 동의 완료 후 다음 단계 진행 가능

### FR-04: 자체 SMS 인증
- 팝업 방식으로 통합본인인증 시스템 연동
- 인증 완료 후 콜백으로 결과 수신
- 인증 성공/실패에 따른 분기 처리
- 인증 실패 시 재시도 안내

### FR-05: 전산플랫폼 연동 (해지 신청)
- REST API (JSON) 방식으로 전산플랫폼 연동
- 고객 동의 + 본인인증 완료 후 **기존 통신사 인터넷 해지 신청** API 호출
- 신청 결과(성공/실패) 처리

### FR-06: 완료 안내 화면
- 해지 신청 완료 후 고객에게 결과 안내
- 성공/실패에 따른 차별화된 안내 메시지

### FR-07: 컨텐츠 관리 API
- 서비스 안내 문구 버전 관리 (DynamoDB 저장)
- 컨텐츠 CRUD API 제공
- CMS 관리자 화면은 향후 개발 (이번 범위: DB + API만)

### FR-08: KTOA 중계기관 연동정보 관리
- KTOA 제공 표준 REST API 연동
- 연동 정보(엔드포인트, 인증키 등) 관리
- 테스트/운영 환경별 연동정보 분리

### FR-09: 고객 공지 관리
- 서비스 점검 안내 등 단순 공지 기능
- 공지 등록/수정/삭제 API
- 고객 화면에서 활성 공지 표시

## 3. Non-Functional Requirements

### NFR-01: 기술 스택
| 구분 | 기술 |
|---|---|
| 프론트엔드 | React (SPA) |
| 백엔드 | AWS Lambda (Python) |
| API | AWS API Gateway (REST) |
| CDN/보안 | CloudFront + WAF |
| 데이터 저장소 | DynamoDB |
| 환경 분리 | 같은 AWS 계정 내 스테이지(dev/prod) 분리 |

### NFR-02: 성능
- URL 유효기간 15분 내 빠른 페이지 로딩
- Lambda cold start 최소화
- 모바일 네트워크 환경 고려

### NFR-03: 가용성
- 서비스 점검 시 공지 표시 기능
- 외부 시스템(본인인증, 전산플랫폼, KTOA) 장애 시 적절한 에러 처리

### NFR-04: 환경 분리
- dev/prod 스테이지 분리 (같은 AWS 계정)
- 환경별 KTOA 연동정보 분리 관리
- 환경별 전산플랫폼 엔드포인트 분리

## 4. 외부 시스템 연동 요약

| 연동 대상 | 방식 | 용도 |
|---|---|---|
| 자체 SMS 인증 | 팝업 (프론트엔드) | 고객 본인인증 |
| 전산플랫폼 | REST API (백엔드) | 기존 통신사 해지 신청 |
| KTOA | REST API (백엔드) | 원스톱전환 중계 |

---

## 5. 프로젝트 범위 요약

### In Scope
- 고객용 모바일 웹 화면 (React SPA)
- 백엔드 API (Lambda + API Gateway + DynamoDB)
- 자체 SMS 인증 팝업 연동
- 전산플랫폼 REST API 연동
- KTOA REST API 연동
- 컨텐츠 관리 DB + API (CMS 화면 제외)
- 고객 공지 관리 API
- CloudFront + WAF 배포

### Out of Scope
- CMS 관리자 웹 화면 (향후 개발)
- 원스톱 전환 ①(신규가입), ③~⑥ 단계 처리
- 보안 확장 규칙 (프로토타입 단계)
- Property-Based Testing
