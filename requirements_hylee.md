# Requirements Document — 원스탑전환 고객동의 서비스

## Intent Analysis

- **User Request**: 원스탑 전환을 위한 고객 동의 웹 화면 구축
- **Request Type**: New Project (Greenfield)
- **Scope Estimate**: Multiple Components (프론트엔드 + 백엔드 + 외부 연동)
- **Complexity Estimate**: Moderate (외부 시스템 연동, 본인인증, 관리자 기능 포함)

## Technology Decisions

| 항목 | 결정 |
|------|------|
| Frontend | React (TypeScript) |
| Backend | Node.js (TypeScript) |
| Database | SQLite |
| Deployment | 로컬 우선, AWS 고려중 |
| 본인인증 연동 | URL Link 방식 (외부 인증 페이지로 리다이렉트) |
| 전산플랫폼 연동 | REST API (JSON) |
| 관리자 UI | 관리자 웹 대시보드 별도 구축 |
| 동시접속 규모 | 대규모 (1,000~10,000명) |
| MMS URL 유효기간 | 24시간 |
| 테스트 방식 | Red-Green-Refactoring TDD |

## Introduction

원스탑 전환 서비스는 KTOA에서 도입한 시스템으로, 인터넷 서비스를 제공하는 모든 사업자에 적용된다. 타 통신사로 인터넷 전환 가입을 원하는 고객에게, 이전 통신사(이전_사업자)가 해지 의사 확인을 위해 MMS를 발송한다. 고객은 MMS 내 URL을 통해 웹 화면에 접속하여 전환 서비스 내용을 확인하고, 본인인증을 완료하여 기존 통신사의 인터넷 해지를 신청한다.

### 원스탑 전환 서비스 전체 흐름

1. 고객이 이후_사업자에 신규 가입을 신청한다
2. 이후_사업자가 관리_기관에 해지를 접수한다
3. 관리_기관이 이전_사업자에 해지 처리를 요청한다
4. **이전_사업자가 고객에게 MMS를 발송하여 해지 의사를 확인한다**
5. **고객이 MMS 내 URL에 접속하여 동의 및 본인인증을 완료한다**
6. 이후_사업자가 고객에게 개통 처리를 수행한다
7. 이전_사업자가 장비 수거 및 해지를 완료한다

본 프로젝트는 **4~5단계**에 해당하는 고객 동의 웹 화면을 구축하는 것이 목적이다.

## Functional Requirements

### FR-1: MMS URL 접속 화면

고객이 이전_사업자가 발송한 MMS 내 URL을 클릭하면 모바일 웹 브라우저에서 Consent_Page를 표시한다.

**Acceptance Criteria:**
1. MMS 내 URL 클릭 시 모바일 웹 브라우저에서 Consent_Page 표시
2. 모바일 환경 최적화 반응형 레이아웃 제공
3. 3초 이내 첫 번째 콘텐츠 렌더링 (FCP)
4. URL 파라미터의 암호화된 토큰으로 고객 식별 및 세션 생성
5. 유효하지 않은 토큰 → 접근 불가 안내 + 서비스 차단
6. 토큰 유효기간(24시간) 만료 → 링크 만료 안내 + 고객센터 연락처
7. HTTPS(TLS 1.2+) 전용 접속
8. 활성 공지사항 존재 시 상단 표시

### FR-2: 원스탑 전환 서비스 안내 화면

고객에게 원스탑 전환 서비스의 내용과 절차를 안내한다.

**Acceptance Criteria:**
1. Consent_Page 접속 시 서비스 개요 표시
2. 처리 절차 단계별 안내 (해지 동의 → 본인인증 → 해지 접수 → 개통 완료 → 자동 해지)
3. 이전_사업자 정보 및 해지 대상 서비스 정보 표시
4. Content_Manager의 최신 안내문구_버전 내용 표시
5. 다음 단계 진행 선택 시 동의 처리 화면으로 이동
6. 스크롤로 전체 내용 확인 가능
7. 안내 문구 로딩 실패 시 기본 안내 문구 표시 + 오류 로그

### FR-3: 원스탑 전환 서비스 신청 동의

고객이 서비스 이용에 필요한 동의 항목을 확인하고 동의한다.

**Acceptance Criteria:**
1. 서비스 안내 확인 완료 후 동의 항목 목록 표시
2. 개인정보 수집·이용 동의, 제3자 제공 동의, 서비스 이용 동의 구분 표시
3. 필수/선택 동의 시각적 구분
4. 동의 항목 상세 내용 팝업/별도 영역 표시
5. 전체 동의 체크박스 → 모든 항목 일괄 선택
6. 필수 동의 완료 + 동의 버튼 → 동의 완료 처리 + 본인인증 진행
7. 필수 미동의 시 → 미동의 항목 강조 + 안내 메시지
8. 동의 내역(항목, 일시, 안내문구_버전) 서버 저장

### FR-4: 본인인증 처리

통합본인인증 시스템(URL Link 방식)을 통해 본인인증을 완료한다.

**Acceptance Criteria:**
1. 동의 완료 후 통합본인인증 URL로 리다이렉트
2. 인증 성공 결과 수신 → 인증 완료 저장 + 해지 신청 단계 진행
3. 인증 실패 → 실패 사유 안내 + 재인증 옵션
4. 120초 타임아웃 → 타임아웃 안내 + 재시도 옵션
5. TLS 1.2+ 암호화 통신
6. 인증 결과(성공 여부, 일시, 수단) 서버 저장
7. 동일 세션 5회 실패 → 인증 차단 + 고객센터 안내

### FR-5: 기존 통신사 인터넷 해지 신청

본인인증 완료 후 전산플랫폼(REST API)을 통해 해지를 신청한다.

**Acceptance Criteria:**
1. 본인인증 완료 → 해지 신청 확인 화면 표시
2. 해지 대상 서비스 정보(이전_사업자명, 서비스 유형, 계약 정보) 표시
3. 해지 신청 확인 → Relay_Connector 통해 전산플랫폼에 REST API 요청
4. 해지 접수 완료 응답 → 완료 메시지 + 접수 번호 표시
5. 해지 접수 실패 → 실패 사유 안내 + 재시도 옵션
6. 30초 타임아웃 → 타임아웃 안내 + 재시도 옵션
7. 해지 신청 이력(접수 번호, 일시, 이전_사업자, 상태) 서버 저장

### FR-6: 고객 완료 안내

해지 신청 완료 후 처리 결과와 향후 절차를 안내한다.

**Acceptance Criteria:**
1. 해지 접수 완료 → 완료 안내 화면 표시
2. 신청 요약 정보(이전_사업자, 이후_사업자, 신청 일시, 접수 번호) 표시
3. 향후 절차 안내 (개통 처리 → 장비 수거 → 해지 완료)
4. 고객센터 연락처 및 문의 방법 표시
5. 화면 닫기 옵션 제공
6. 완료 처리 이력(완료 일시, 최종 상태) 서버 저장

### FR-7: 컨텐츠 및 안내문구 관리 (관리자)

관리자가 서비스 안내 문구를 관리한다.

**Acceptance Criteria:**
1. 안내 문구 등록, 수정, 삭제 기능
2. 안내문구_버전 자동 부여 + 버전 이력 관리
3. 새 문구 등록 시 새 버전 생성 + 이전 버전 보관
4. 동의 항목별 컨텐츠 개별 관리
5. 특정 버전 활성화 → 즉시 반영
6. 버전별 생성/수정 일시, 활성화 상태, 작성자 저장
7. 활성 버전 미존재 → 기본 문구 제공 + 관리자 알림
8. 고객 동의 시점의 안내문구_버전을 동의 이력과 함께 저장

### FR-8: 중계기관 연동정보 관리 (관리자)

관리자가 타통신사 연동을 위한 중계기관 연동정보를 관리한다.

**Acceptance Criteria:**
1. 중계기관별 연동정보(인증키, 엔드포인트 URL, 타임아웃) 등록/수정/조회
2. 인증키 암호화 저장
3. 연동정보 수정 시 변경 이력 저장
4. 이전_사업자별 연동정보 구분 관리
5. Transfer_Processor 요청 시 해당 이전_사업자 연동정보 사용
6. 인증키 만료 → 관리자 알림 + 오류 로그
7. 연동 요청/응답 이력 저장
8. 관리자 연동 테스트 요청 → 테스트 전송 + 상태 반환

### FR-9: 테스트/운영 환경 분리 관리

테스트환경과 운영환경을 분리하여 관리한다.

**Acceptance Criteria:**
1. 테스트/운영 환경 설정 정보 분리 관리
2. 환경별 독립 연동정보 관리
3. 환경별 독립 DB 접속 정보 관리
4. 환경 변수로 실행 환경 식별 + 해당 설정 로드
5. 운영환경 설정을 테스트환경에서 조회/수정 차단
6. 환경 식별 실패 → 애플리케이션 시작 중단 + 오류 로그
7. 환경별 설정 변경 이력 저장

### FR-10: 고객 공지 관리 (관리자)

관리자가 고객에게 표시할 공지사항을 관리한다.

**Acceptance Criteria:**
1. 공지사항 등록, 수정, 삭제, 조회
2. 공지별 제목, 내용, 게시 시작/종료일, 활성화 상태 관리
3. 등록 시 게시 시작/종료일 필수 입력
4. 게시 기간 내 + 활성화 → Consent_Page 상단 표시
5. 게시 종료일 경과 → 자동 비활성화
6. 복수 공지 동시 활성 → 등록일 최신순 정렬
7. 공지 유형(긴급/일반/안내) 구분 + 유형별 시각적 스타일
8. 공지 변경 이력 저장

## Non-Functional Requirements

### NFR-1: 성능
- FCP(First Contentful Paint) 3초 이내
- 동시 접속 1,000~10,000명 지원
- 전산플랫폼 API 응답 타임아웃: 30초
- 본인인증 타임아웃: 120초

### NFR-2: 보안
- HTTPS(TLS 1.2+) 전용
- URL 토큰 암호화 + 24시간 유효기간
- 인증키 암호화 저장
- 본인인증 5회 실패 시 세션 차단

### NFR-3: 가용성
- 테스트/운영 환경 완전 분리
- 환경 식별 실패 시 안전한 시작 중단
- 안내 문구 로딩 실패 시 기본 문구 폴백

### NFR-4: 테스트
- Red-Green-Refactoring TDD 방식 적용
- 모든 비즈니스 로직에 단위 테스트 작성
