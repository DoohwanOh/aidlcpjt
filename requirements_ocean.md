# 요구사항 정의서: 원스탑전환 고객동의 서비스 구축
 
## Intent Analysis
 
- **사용자 요청**: 원스톱 전환을 위한 고객 동의 웹화면 구축

- **요청 유형**: New Project (신규 프로젝트)

- **범위 추정**: Multiple Components (프론트엔드 + 백엔드 + 외부 연동)

- **복잡도 추정**: Moderate (외부 시스템 연동 포함)
 
## 1. 프로젝트 개요
 
### 1.1 프로젝트명

원스탑전환 고객동의 서비스 구축
 
### 1.2 한 줄 요약

타 통신사로 인터넷 전환 가입을 원하는 고객이 이전 통신사로부터 수신한 MMS를 통해 접속하여, 전환 서비스 내용 확인 및 본인인증 후 기존 통신사 인터넷 해지를 신청하는 웹 화면 구축
 
### 1.3 배경 및 문제 정의

- KTOA에서 도입한 원스톱 전환 서비스 지원을 위해 고객 동의 웹화면 필요

- 타 통신사로 인터넷 전환 가입을 원하는 고객에게 **이전 통신사가 MMS를 발송**하고, 고객은 해당 MMS 내 URL로 접속

- 고객은 웹 화면에서 전환 서비스 내용을 확인하고, 본인인증을 완료하여 **기존 통신사의 인터넷 해지를 신청**

- 기존: 약정 가입 → LGU+ 신규신청 → LGU+ 개통완료 → 이전통신사 해지접수 → 해지완료 (5단계)

- 신규(원스톱): 원스톱전환서비스 → LGU+ 신규가입 + 이전 통신사 해지 신청 → LGU+ 개통완료 후 이전 통신사 자동 해지 (3단계)
 
### 1.4 서비스 흐름 (원스톱 전환)

1. 고객 → 여후 사업자(LGU+): 신규 가입 신청

2. 여후 사업자 → KTOA(관리 기관): 해지 접수

3. KTOA → 이전 사업자: 해지 처리 요청

4. 이전 사업자 → KTOA: 해지 의사 확인

5. 여후 사업자: 개통 처리

6. 이전 사업자: 장비 수거 / 해지 완료
 
### 1.5 상담 플로우

첫인사 → 목적전달 & 관심유발 → 상품안내 및 가입유도 → 원스탑 전환여부 탐색 → 타사 가입정보 탐색 → 본인인증 및 동의 → 상담종료
 
## 2. 기능 요구사항 (Functional Requirements)
 
### FR-001: MMS URL 접속 화면

- **설명**: 이전 통신사가 고객에게 발송한 MMS 내 URL을 통해 접속하는 웹 랜딩 페이지

- **상세**:

  - 모바일 브라우저 최적화 (반응형)

  - URL 유효성 검증 (만료, 중복 접속 방지)

  - 서비스 안내 콘텐츠 표시
 
### FR-002: 원스톱 전환 서비스 안내 화면

- **설명**: 고객이 전환 서비스 내용을 확인할 수 있는 안내 화면

- **상세**:

  - 서비스 설명 및 절차 안내

  - 안내 문구 버전 관리 (백엔드 컨텐츠 관리)

  - 고객 공지사항 표시
 
### FR-003: 원스톱 전환 서비스 신청 동의

- **설명**: 고객이 기존 통신사 인터넷 해지에 동의하는 화면

- **상세**:

  - 동의 항목 표시 (필수/선택 구분)

  - 동의 내역 저장

  - 전체 동의 / 개별 동의 기능
 
### FR-004: 통합본인인증 연동

- **설명**: 고객 본인 확인을 위한 통합본인인증 시스템 연동

- **상세**:

  - URL Link 방식으로 본인인증 페이지 연동

  - 인증 완료 결과 수신 및 처리

  - 인증 실패 시 재시도 안내

  - 본인인증 완료 후 해지 신청 프로세스 진행
 
### FR-005: 전산플랫폼 연동 (기존 통신사 인터넷 해지 신청)

- **설명**: 본인인증 완료 후 기존 통신사 인터넷 해지를 신청하기 위한 전산플랫폼 REST API 연동

- **상세**:

  - REST API 호출을 통한 해지 신청

  - 요청/응답 처리 및 에러 핸들링

  - 처리 결과 고객 안내
 
### FR-006: KTOA 중계기관 연동

- **설명**: 타통신사 연동을 위한 KTOA 중계기관 REST API 연동

- **상세**:

  - REST API 기반 연동 정보 관리

  - 해지 접수 및 상태 조회

  - 연동 정보 테스트/운영 환경 분리 관리
 
### FR-007: 고객 완료 안내

- **설명**: 모든 처리 완료 후 고객에게 결과 안내

- **상세**:

  - 신청 완료 안내 화면

  - 처리 상태 요약 표시

  - 후속 절차 안내
 
### FR-008: 컨텐츠 관리 (백엔드)

- **설명**: 페이지 구성을 위한 컨텐츠 관리 기능

- **상세**:

  - 서비스 안내 문구 버전 관리

  - 고객 공지 관리

  - SQLite 기반 데이터 저장 (로컬 환경, 이후 AWS 전환 검토)
 
### FR-009: 연동 정보 관리 (백엔드)

- **설명**: 타통신사 연동을 위한 중계기관 연동 정보 관리

- **상세**:

  - KTOA 연동 설정 관리

  - 테스트환경/운영환경 분리 관리

  - 연동 상태 모니터링

# 유저 스토리: 원스톱전환 고객동의 서비스
 
## 고객 여정 맵
 
```

MMS URL 접속 → 서비스 안내 확인 → 동의 처리 → 본인인증 → 해지 신청 → 완료 안내

```
 
---
 
## Epic 1: 고객 웹 화면 (전환 고객 여정)
 
### US-001: MMS URL 접속

**As a** 전환 고객,

**I want to** MMS에 포함된 URL을 클릭하여 웹 화면에 접속하고 싶다,

**So that** 원스톱 전환 서비스를 시작할 수 있다.
 
**Priority**: P1 (최우선)

**Persona**: 전환 고객 (김민수)
 
**Acceptance Criteria**:

- Given 고객이 유효한 MMS URL을 클릭했을 때,

  When 모바일 브라우저에서 페이지가 로드되면,

  Then 원스톱 전환 서비스 랜딩 페이지가 표시된다.

- Given 만료되거나 유효하지 않은 URL로 접속했을 때,

  When 페이지가 로드되면,

  Then 적절한 안내 메시지가 표시된다.
 
---
 
### US-002: 서비스 안내 확인

**As a** 전환 고객,

**I want to** 원스톱 전환 서비스의 내용과 절차를 확인하고 싶다,

**So that** 서비스를 이해하고 동의 여부를 판단할 수 있다.
 
**Priority**: P1

**Persona**: 전환 고객 (김민수)
 
**Acceptance Criteria**:

- Given 고객이 랜딩 페이지에 접속했을 때,

  When 서비스 안내 화면이 표시되면,

  Then 전환 서비스 설명과 절차가 명확하게 안내된다.
 
---
 
### US-003: 서비스 신청 동의

**As a** 전환 고객,

**I want to** 원스톱 전환 서비스 이용에 동의하고 싶다,

**So that** 본인인증 및 해지 신청 절차를 진행할 수 있다.
 
**Priority**: P1

**Persona**: 전환 고객 (김민수)
 
**Acceptance Criteria**:

- Given 고객이 서비스 안내를 확인한 후,

  When 동의 항목(필수/선택)을 선택하고 동의 버튼을 누르면,

  Then 동의 내역이 저장되고 본인인증 단계로 이동한다.

- Given 필수 동의 항목을 선택하지 않았을 때,

  When 동의 버튼을 누르면,

  Then 필수 항목 동의 안내 메시지가 표시된다.
 
---
 
### US-004: 본인인증 처리

**As a** 전환 고객,

**I want to** 본인인증을 완료하고 싶다,

**So that** 기존 통신사 인터넷 해지를 신청할 수 있다.
 
**Priority**: P1

**Persona**: 전환 고객 (김민수)
 
**Acceptance Criteria**:

- Given 고객이 동의를 완료한 후,

  When 본인인증 URL Link로 이동하여 인증을 완료하면,

  Then 인증 결과가 수신되고 해지 신청 단계로 진행된다.

- Given 본인인증에 실패했을 때,

  When 인증 결과가 실패로 반환되면,

  Then 재시도 안내 메시지가 표시된다.
 
---
 
### US-005: 기존 통신사 해지 신청

**As a** 전환 고객,

**I want to** 기존 통신사의 인터넷 해지를 신청하고 싶다,

**So that** 원스톱으로 전환 절차를 완료할 수 있다.
 
**Priority**: P1

**Persona**: 전환 고객 (김민수)
 
**Acceptance Criteria**:

- Given 본인인증이 완료된 후,

  When 전산플랫폼 REST API를 통해 해지 신청이 처리되면,

  Then 해지 신청 결과가 고객에게 표시된다.

- Given 해지 신청 API 호출이 실패했을 때,

  When 타임아웃 또는 에러가 발생하면,

  Then 에러 안내 메시지가 표시된다.
 
---
 
### US-006: 완료 안내

**As a** 전환 고객,

**I want to** 모든 처리가 완료된 결과를 확인하고 싶다,

**So that** 해지 신청이 정상 처리되었음을 알 수 있다.
 
**Priority**: P1

**Persona**: 전환 고객 (김민수)
 
**Acceptance Criteria**:

- Given 해지 신청이 완료된 후,

  When 완료 안내 화면이 표시되면,

  Then 처리 결과 요약과 후속 절차 안내가 표시된다.

## Epic 2: 관리자 기능
 
### US-007: 관리자 인증 (로그인)

**As a** 서비스 관리자,

**I want to** 관리 시스템에 인증(로그인)하고 싶다,

**So that** 권한이 있는 사용자만 컨텐츠 및 연동 정보를 관리할 수 있다.
 
**Priority**: P2

**Persona**: 서비스 관리자 (박지영)
 
**Acceptance Criteria**:

- Given 관리자가 관리 시스템에 접속했을 때,
